# B11 blocker — Navigator removal is non-mechanical

**Date:** 2026-05-26
**Branch:** `hook-inject` (worktree `worktree-agent-a859fc349634a47c2`, fast-forwarded onto `3261c79e6d`)
**Task:** B11 — register `ConsolePanel`, delete `TerminalPanel` + `SolutionSessionsNavigator`.

## TL;DR

Task B11 as written assumes the deletions of `crates/terminal_view/src/terminal_panel.rs`
and `crates/solution_agent/src/navigator.rs` are mechanically resolvable from the
compile-error list. They are not. Two large refactors hide behind them that the
plan never spells out:

1. **`SolutionSessionView` is structurally coupled to `SolutionSessionsNavigator`.**
   The view holds a `WeakEntity<SolutionSessionsNavigator>` field and calls
   `navigator.render_status_row(...)` from inside its own `render` method
   (`crates/solution_agent/src/session_view.rs:2585`). The status row owns the
   model selector, token meter, "Thinking… 3m12s" timer, compact button, and
   history popover — i.e. all the chrome users expect above the compose box.
   Deleting `navigator.rs` (1493 lines) implies also deleting `status_row.rs`
   (1112 lines) and `compact.rs` (449 lines), then re-implementing the entire
   status row as a free-function or moving its state into ConsolePanel.
   ChatProvider's current trick of passing `WeakEntity::new_invalid()` makes
   it compile-clean today, but it silently drops the entire status row in
   ConsolePanel chat tabs — *a regression we'd ship if we proceed*.

2. **`TerminalPanel` callsites in upstream crates use it as a *factory*, not a
   panel.** ~32 external references across `agent_ui` (4 files), `debugger_ui`
   (2 files), `run_config_ui`, `vim`, `command_palette`, `workspace`, and `zed`
   itself. The hot APIs are `TerminalPanel::add_center_terminal`,
   `TerminalPanel::new_terminal` (5+ unique callsites for LSP run, dap, run-config,
   spawn-task, hover "Run in terminal"). `ConsolePanel::add_terminal_tab(cwd)`
   today only accepts a CWD — none of the `SpawnInTerminal` / `TaskState` /
   reveal-strategy / debug-terminal plumbing those callers need. Either we
   port that API surface onto `ConsolePanel` (large), or every caller gets
   stubbed/disabled (silent UX regression on LSP-run-in-terminal, dap-attach,
   etc.).

`grep -rln "SolutionSessionsNavigator" crates/ | wc -l` → **9 files**, of which 6

```rust
// its own chrome and does not use SolutionSessionsNavigator.
let navigator: WeakEntity<SolutionSessionsNavigator> = WeakEntity::new_invalid();
```

## Required design call

Three options, all needing user approval:

**A. Lift status row out of Navigator (recommended).**
   - Move `render_status_row` + activity-tick timer + agent/model/working-dir
     resolution out of `SolutionSessionsNavigator` into a free function
     `solution_agent::status_row::render(session_id, view, cx) -> AnyElement`.
   - Drop the `navigator: WeakEntity<...>` field from `SolutionSessionView`;
     have `render` call the free function directly.
   - Delete `navigator.rs` + `compact.rs` (move `Compact` action handler onto
     `SolutionSessionView`).
   - Effort: ~2 days. Preserves UX exactly.

**B. Keep Navigator alive, just don't register it as a Panel.**
   - Don't delete `navigator.rs` at all in B11. Drop only its
     `impl Panel for SolutionSessionsNavigator` registration in
     `initialize_panels`. Plumb a real (non-invalid) `Entity<Navigator>` into
     each ConsolePanel chat tab — e.g. ConsolePanel owns one shared Navigator
     entity solely to host status row state.
   - Effort: ~half a day. Status row works. But Navigator's tab-list /
     selection-state / db persistence overlaps with ConsolePanel's, so it's
     dead-state-on-paper; risk of resurrected dual UI later.

**C. Accept the regression.**
   - Land B11 with `WeakEntity::new_invalid()` everywhere. Status row,
     compact action, history popover all silently disappear from chat tabs
     in ConsolePanel. Open a follow-up issue.
   - Effort: ~half a day to clean up, but ships visibly broken chat UX.

`TerminalPanel` factory-API replacement (B11 step 5) is independent of the
above and also large (~32 callsites). The `ConsolePanel::add_terminal_tab`
API needs at minimum: take a `SpawnInTerminal` / `TaskState`, accept a
`RevealStrategy`, expose pane focus, support debug terminals. That's a
distinct ~1-day port.

Pause B11. Pick option A or B for the navigator question. Once chosen, B11
becomes ~1 commit (panel wireup + `init` call) + a separate commit for the
chosen navigator refactor + a separate commit for the TerminalPanel factory
port. Trying to land it as one batch will produce a 30+ file churn that's
hard to review and easy to break.
