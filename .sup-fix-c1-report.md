Branch: `chat-supervisor` · crate: `solution_agent` (+ `workspace_events`)

## Guard placement (covers C1 + I1)

Rather than one literal call site, the two suppressed side effects live in two
methods, so the guard is applied at each — but BOTH key on the same predicate
(`is_supervisor_ephemeral`) and both leave internal bookkeeping untouched.

### C1 — workspace wire emit
`crates/solution_agent/src/store.rs` `emit_session_state_changed_workspace`
(now ~line 6727): after the `self.sessions.get(session_id)` lookup, added

```rust
if entity.read(cx).is_supervisor_ephemeral {
    return;
}
```

This is the single funnel for `workspace.session_state_changed`. It is called
from `mark_state_changed`, which still runs `cx.emit(SessionStateChanged)` (the
INTERNAL event) and bumps `state_seq` — only the sequenced wire emit forwarded to
mobile by `remote_control`'s allow-list (`should_forward_event` forwards all
`workspace.*`) is dropped for the hidden judge/auditor.

### I1 — desktop notification
`crates/solution_agent/src/store.rs` `mutate_state` (~line 6800): the
`decide_notification` branch is now gated:

```rust
let is_supervisor_ephemeral = session.read(cx).is_supervisor_ephemeral;
if !is_supervisor_ephemeral
    && let Some(decision) = notifier::decide_notification( ... ) { ... }
```

So an Errored judge or a judge turn crossing `JUDGE_TIMEOUT_SECS` no longer pops
a tray toast. All bookkeeping ABOVE this branch (state transition, Stopping
safety-net drop, inline-subagent GC) still runs for the ephemeral session.

**Why this covers both and is safe:** the only suppressed things are the two
OUTWARD side effects of the ephemeral session's own state flips (mobile wire
event + desktop toast). The internal `SessionStateChanged` cx.emit and `state_seq`
watermark are preserved — `message_generator` subscribes to `SessionStateChanged`
to detect the judge going Idle/Errored, so suppressing it would break judge
completion.

## Judge-teardown does NOT depend on the suppressed events (confirmed)

`tick_supervisor` (`store.rs` ~1663) detects judge-end purely via the `Judging`
status timeout (`stuck_ms >= JUDGE_TIMEOUT_SECS * 1000 → on_judge_failed`); the
verdict tool tears the judge down on the happy path. The in-code comment states
this is done "without per-judge session-state subscriptions." Neither the wire
emit nor the desktop notifier feeds the watchdog, so the guard cannot break
teardown.
